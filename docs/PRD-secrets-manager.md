# PRD. Secrets Manager (비밀번호 보관함)

> 특정 사이트의 비밀번호를 `사이트 → (카테고리) → 비밀번호` 구조로 보관하는 1인용 암호화 보관함. Cloudflare로 공개 호스팅하되 Cloudflare Access로 앞단을 차단하고, 6자리 PIN으로 접속한다.

---

## 1. 요약

Secrets Manager는 은행·페이·쇼핑 같은 사이트의 비밀번호를 한곳에 암호화해 보관하는 본인 한 명용 웹 보관함이다. 사이트를 만들고 그 아래에 비밀번호를 직접 두거나(예. 네이버페이 → 결제비밀번호), 필요하면 카테고리를 한 단계 끼워 정리한다(예. 우리은행 → 계좌번호 → 비밀번호). 접속은 6자리 PIN으로 하고, 공개 노출은 Cloudflare Access가 1차로 막으며, 실제 비밀번호 본문은 별도 마스터에서 도출한 키로 AES-256-GCM 암호화한다.

## 2. 담당자

| 이름 | 역할 | 코멘트 |
|------|------|--------|
| 사용자 D | 제품 오너 · 유일한 사용자 | 공개 호스팅(Cloudflare) + 6자리 PIN 접속, 비밀번호 단일 도메인을 결정한다. |
| (미정) | 엔지니어링 | Next.js·NestJS·PostgreSQL 스택을 구현·운영하고 단일 서버에 배포한다. |

> 본 제품은 1인 사용을 전제한다. 담당자 표는 향후 협업 확장에 대비한 자리만 둔다.

## 3. 배경

은행·카드·증권·페이·쇼핑몰의 비밀번호와 그에 딸린 번호(계좌번호, 결제비밀번호 등)가 머릿속과 메모 앱에 흩어져 있다. 범용 비밀번호 관리자는 무겁고, 메모 앱은 평문이라 위험하다. 필요한 건 사이트 단위로 단순하게 정리하면서도 본문은 확실히 암호화되는 가벼운 개인 보관함이다.

기존 Secrets Manager는 여러 도메인을 다루다 비밀번호 보관함, 모바일 UI, PWA까지 쌓아 왔다. 그러나 실제로 매일 필요한 것은 비밀번호 관리 하나였다. 그래서 다른 도메인을 모두 덜어내고, 비밀번호 관리에만 집중하는 단일 제품으로 다시 세운다.

**왜 지금인가.** 이미 갖춰진 NestJS·Prisma·Next.js·암호화(Argon2id + AES-256-GCM)·모바일 UI 자산을 비밀번호 관리에 그대로 재사용할 수 있고, Cloudflare로 도메인을 등록해 어디서나 접속하려는 새 요구가 생겼기 때문이다.

---

## 4. 목표

**목표.** 한 명의 사용자가 사이트별 비밀번호를 어디서나 안전하게 보관·열람하되, 공개 인터넷에 노출되더라도 PIN·Cloudflare Access·암호화의 다층 방어로 보호받게 한다.

**왜 중요한가.** 흩어진 비밀번호를 한곳에 암호화해 두면 분실·유출 위험이 줄고, 사이트 단위 구조 덕분에 빠르게 찾을 수 있다.

**비전·전략 정합성.** 무겁지 않은 1인용 도구로, 본인이 통제하는 단일 서버에서 운영하는 프라이버시 우선 방향에 부합한다.

### 핵심 성과 지표 (Key Results)

| KR | 측정 방식 | 목표 |
|----|----------|------|
| 평문 미저장 | DB에 저장되는 비밀번호 본문 중 암호화되지 않은 건수 | 0건 |
| 무단 접근 차단 | Cloudflare Access를 통과하지 못한 외부 요청이 앱에 도달하는 비율 | 0% |
| 무차별 대입 방어 | 6자리 PIN 5회 실패 후 추가 시도 차단 동작 | 100% backoff 발동 |
| 열람 속도 | 마스터 해제 상태에서 비밀번호 1건 복호화·표시 시간 | 200ms 이내 |
| 데이터 통제 | 비밀번호 본문에 접근 가능한 외부 제3자(매니지드 서비스) 수 | 0 (단일 자가 서버) |

## 5. 시장 세그먼트

**누구를 위해 만드는가.** 다음 일(job)을 가진 단일 사용자다.

- 사이트별 비밀번호와 결제비밀번호·계좌번호를 한곳에 정리해 두고 싶다.
- 그 본문이 평문으로 떠돌지 않고 확실히 암호화되길 원한다.
- 집·회사·외부 어디서든 내 도메인으로 접속하고 싶지만, 공개 노출 위험은 최소화하고 싶다.
- 매니지드 클라우드에 데이터를 맡기지 않고 내가 운영하는 서버 한 대에 두고 싶다.

**제약.**

- 단일 사용자, 한국어 환경.
- 공개 도메인으로 접근하되 Cloudflare Access가 앞단을 차단한다.
- 비밀번호 본문은 반드시 암호화한다.

> 시장은 인구 통계가 아니라 사람의 할 일로 정의한다. 여기서는 사이트별 비밀번호를 안전하게 보관·열람한다는 단일 job이 핵심이다.

## 6. 가치 제안

| 고객의 일(Job) | 얻는 것 | 피하는 고통 | 경쟁 대비 우위 |
|----------------|---------|-------------|----------------|
| 사이트별 비밀번호 정리 | 사이트→(카테고리)→비밀번호의 단순 2~3계층 | 메모 앱·머릿속에 흩어진 번호 | 범용 관리자보다 가볍고 사이트 중심 |
| 본문 보호 | AES-256-GCM 암호화, 마스터 키는 메모리에만 | 평문 메모 유출 | 마스터·PIN 분리로 PIN 유출만으론 본문 안전 |
| 어디서나 접속 | 내 도메인 + 6자리 PIN | VPN·포트 개방의 번거로움 | Cloudflare Tunnel로 개방 없이 외부 접속 |
| 공개 노출 방어 | Cloudflare Access 1차 차단 + PIN 2차 | 공개 보관함가 PIN 하나로만 노출 | 다층 방어로 온라인 대입 위험 감소 |
| 데이터 통제 | 단일 자가 서버에 app+DB | 제3자 인프라 의존 | 매니지드 미사용, 본문은 어차피 암호화 |

**가치 곡선.** 경쟁 제품이 다기능·다인·자동입력으로 경쟁할 때, 이 제품은 단순한 사이트 구조·강한 암호화·자가 호스팅 통제에 자원을 집중한다.

## 7. 솔루션

### 7.1 사용자 흐름

```
[인터넷] → Cloudflare Access(이메일 OTP 등 1차 차단)
         → 6자리 PIN 로그인(서버측 해시 + rate limit/backoff)
         → 세션(SameSite=Strict 쿠키)
         → 사이트·비밀번호 목록 조회(라벨은 평문이라 검색 가능)
         → 비밀번호 열람·추가·수정 시 마스터 입력 → 보관함 해제(AES 키 메모리 보관)
         → idle N분 후 마스터 키 폐기(재잠금), PIN 세션은 유지
```

### 7.2 핵심 기능

**F1. PIN 설정·로그인·잠금**

- 첫 실행 시 6자리 PIN을 설정한다. PIN은 Argon2id로 해시해 저장하며 절대 암호화 키로 쓰지 않는다.
- 이후 접속은 PIN으로 로그인한다. 5회 연속 실패 시 60초 backoff를 발동한다.
- Cloudflare Access가 앞단에서 신원을 1차 차단하므로 PIN은 2차 방어다.

**F2. 마스터로 보관함 해제/잠금**

- 비밀번호 본문을 열람·추가·수정하려면 마스터를 입력한다. 마스터는 Argon2id로 암호화 키를 도출하고 키는 서버 메모리에만 둔다.
- idle N분(권장 5분) 경과 시 키를 폐기해 자동 잠근다. 잠금 상태에서 본문 접근은 거부한다.
- 마스터 정확성은 verifyToken(고정 평문의 AEAD)으로 검증한다.

**F3. 사이트 CRUD**

- 사이트를 라벨(예. 우리은행, 네이버페이)과 선택적 아이콘으로 등록·수정·삭제한다.
- 사이트 라벨은 검색용 평문이다. 사이트 삭제 시 하위 카테고리·비밀번호도 함께 삭제한다(확인 다이얼로그).

**F4. 카테고리 CRUD (선택적)**

- 사이트 하위에 카테고리를 한 단계 둘 수 있다(예. 우리은행 → 계좌번호). 카테고리 없이 사이트 직속 비밀번호도 허용한다.
- 카테고리는 enum이 아니라 자유 입력 라벨이다. 계층은 최대 1단계로 제한한다.

**F5. 비밀번호 CRUD**

- 비밀번호 항목을 라벨 + 본문으로 등록한다. 본문은 AES-256-GCM으로 암호화해 ciphertext·iv·authTag로 저장한다.
- 열람 시 마스터 해제 상태에서 복호화해 표시하고, 복사 버튼은 30초 후 클립보드를 자동 클리어한다.
- 항목은 사이트 직속이거나(categoryId=null) 카테고리 하위에 속한다.

**F6. 검색**

- 사이트·카테고리·비밀번호 라벨(평문)에 대해 부분 일치 검색한다. 본문은 검색 대상이 아니다.

### 7.3 데이터 모델

| 모델 | 핵심 필드 | 비고 |
|------|----------|------|
| `Site` | id, label, icon?, createdAt | 이름 중심. 라벨은 평문 |
| `Category` | id, siteId(FK), label | 선택적. 사이트당 0개 이상, 1계층 |
| `Secret` | id, siteId(FK), categoryId?(nullable FK), label, ciphertext, iv, authTag, kdfVersion, createdAt, updatedAt | 본문 암호화. categoryId=null이면 사이트 직속 |
| `PinCredential` | id, pinHash, failCount, lockUntil | 6자리 PIN 로그인 |
| `VaultMaster` | id="singleton", kdfParams, verifyToken, kdfVersion | 암호화 마스터(기존 vault 개념 재사용) |

예시 매핑.
- 우리은행 → 계좌번호(Category) → 비밀번호(Secret, categoryId 지정)
- 우리은행 → 결제비밀번호(Secret, categoryId=null)
- 네이버페이 → 결제비밀번호(Secret, categoryId=null)

### 7.4 기술

- **모노레포.** pnpm workspace(`apps/web`, `apps/api`).
- **프런트엔드.** Next.js 15 App Router + React 19. 모바일 우선 레이아웃·PWA·접근성은 기존 자산 재사용. 라우트는 비밀번호 중심으로 축소.
- **백엔드.** NestJS 10 + Prisma. PIN/마스터 인증, 암호화, 사이트·카테고리·비밀번호 REST 엔드포인트.
- **저장소.** **PostgreSQL.** Prisma `postgresql` 프로바이더, `DATABASE_URL` 연결, 커넥션 풀. 백업은 `pg_dump`.
- **암호화.** Argon2id(m=64MiB, t=3, p=1, salt 16바이트) KDF + AES-256-GCM(nonce 12바이트, tag 16바이트). KDF 버전닝으로 강화 경로 확보.
- **앞단·인증.** Cloudflare(도메인 등록 + DNS + Access + Tunnel). 앱·DB는 직접 운영하는 단일 서버에 함께 둔다. CORS·Origin은 내 도메인만 허용.

### 7.5 배포

- **단일 서버 한 대**에 web·api·postgres를 함께 세팅한다(예. `docker-compose`로 web + api + postgres + cloudflared 한 묶음).
- `cloudflared` Tunnel이 바깥으로 연결을 열어 포트 개방·고정 IP 없이 외부 접속을 가능하게 한다.
- Cloudflare Access를 도메인에 적용해 앱 도달 전에 신원을 차단한다. **Access 설정 누락은 배포 차단 사유**다.
- 매니지드 서비스(매니지드 Postgres·PaaS)는 사용하지 않는다.

### 7.6 가정

- 사용자는 1명, 비밀번호 항목은 수백~수천 건 규모다.
- 단일 서버는 상시 가동되며 사용자가 OS·DB 백업을 책임진다.
- Cloudflare Access가 항상 활성화되어 있다(미설정 시 위협 모델 붕괴).
- 마스터를 분실하면 본문 복구가 불가하므로 setup 직후 암호화 백업 1회 안내를 제공한다.

## 8. 릴리스

날짜 대신 상대적 단계로 표현한다.

| 단계 | 범위 | 비고 |
|------|------|------|
| 0차 (정리) | 비밀번호 외 도메인 제거, 문서 피벗, Prisma를 PostgreSQL로 전환 | 본 PRD 기준 |
| 1차 (P0) | PIN 설정·로그인·backoff, 마스터 해제/잠금, 사이트·카테고리·비밀번호 CRUD, 검색 | 핵심 보관함 |
| 2차 | Cloudflare Tunnel + Access 배포, 암호화 백업·복원 | 공개 접속 확정 |
| 후속 | 마스터 변경(re-key) UI, 항목 정렬·태그, 가져오기/내보내기 강화 | 필요 시 |

**1차에 넣지 않는 것.** 다중 사용자·회원가입, 브라우저 자동입력, 클라우드 동기화, 매니지드 인프라, 마스터 변경 UI(KDF 마이그레이션 로직만 보존).

## 부록. 용어 정의

| 용어 | 설명 |
|------|------|
| 사이트 (Site) | 비밀번호를 묶는 최상위 단위. 라벨 + 선택적 아이콘. |
| 카테고리 (Category) | 사이트 하위의 선택적 1계층 묶음(예. 계좌번호). 자유 입력 라벨. |
| 비밀번호 (Secret) | 라벨 + 암호화된 본문 1건. 사이트 직속이거나 카테고리 하위. |
| PIN | 6자리 접속용 비밀번호. Argon2id로 해시 저장. 암호화 키로는 쓰지 않음. |
| 마스터 | 비밀번호 본문 암호화 키를 도출하는 별도 비밀. 메모리에만 존재. |
| KDF / AEAD | 키 도출 함수(Argon2id) / 인증 암호화(AES-256-GCM). |
| verifyToken | 마스터 정확성 검증용 고정 평문의 AEAD 출력. |
| Cloudflare Access | 앱 도달 전 신원을 차단하는 Zero Trust 앞단 인증. |
| Cloudflare Tunnel | 서버가 바깥으로 연결을 열어 포트 개방 없이 외부 접속을 가능하게 하는 터널. |
| Unlock / Lock 상태 | 마스터 키가 메모리에 있는 상태 / 폐기된 상태. |
