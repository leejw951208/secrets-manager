# Spec. 결제 리마인더

> 예정일이 임박했거나 지난 정기 지출을 대시보드와 리스트에서 즉시 알아보고 놓치기 전에 처리하게 하는 앱 내부 리마인더 기능.

## 배경

정기 지출 관리의 핵심 가치는 결제 예정일과 금액을 미리 알고 대응하는 데 있다. 현재 앱은 대시보드의 다음 7일 예정, 캘린더, 결제 리스트를 통해 예정 항목을 조회할 수 있지만, 사용자가 직접 화면을 훑어야만 긴급도를 알 수 있다. 예정일이 오늘인지, 3일 이내인지, 이미 지났는지의 차이가 시각적으로 충분히 강하지 않다.

이번 기능은 외부 푸시 알림이나 백그라운드 스케줄러 없이, 로컬 앱 내부에서 결제 임박도와 연체 상태를 명확히 드러내는 첫 단계다. 기존 `ExpenseOccurrence` 데이터와 `GET /occurrences` API를 재사용해 구현 범위를 작게 유지하고, 결제 완료 처리 흐름으로 자연스럽게 이어지게 한다.

## 기능 목록

### 1. 대시보드 리마인더 요약

대시보드 상단 또는 기존 요약 카드 근처에 오늘, 3일 이내, 7일 이내, 연체 예정 항목 수를 보여준다. 사용자는 앱을 열자마자 가장 급한 결제 상태를 확인할 수 있다.

**동작 방식**

- 대시보드는 오늘 기준으로 `SCHEDULED` 상태의 occurrence를 조회한다.
- 예정일이 오늘보다 이전이면 `연체`로 분류한다.
- 예정일이 오늘이면 `오늘`, 오늘 이후 3일 이내면 `3일 이내`, 오늘 이후 7일 이내면 `7일 이내`로 분류한다.
- 각 요약 항목은 클릭 시 `/occurrences`로 이동하며 해당 기간과 상태 필터가 적용된다.

**포함 범위**

- 앱 내부 배지와 카드 형태의 리마인더.
- `SCHEDULED` occurrence 기준의 임박도 계산.
- 기존 `/occurrences` 화면으로 이어지는 필터 링크.

**제외 범위**

- OS/browser push notification. 이유는 권한, 서비스 워커, 백그라운드 실행 정책이 얽혀 첫 리마인더 기능 범위를 과하게 키운다.
- 이메일, 문자, 외부 캘린더 연동. 이유는 로컬 전용 원칙과 외부 네트워크 호출 없음 제약을 깨뜨린다.

### 2. 결제 리스트 긴급도 표시

`/occurrences` 리스트에서 각 항목의 긴급도를 배지와 정렬 힌트로 보여준다. 같은 리스트 안에서도 오늘 처리해야 할 항목과 아직 여유 있는 항목을 구분한다.

**동작 방식**

- 각 occurrence 행에 `연체`, `오늘`, `3일 이내`, `7일 이내`, `예정` 중 하나의 리마인더 배지를 표시한다.
- `PAID`와 `SKIPPED` 항목은 긴급 리마인더 배지를 표시하지 않는다.
- 기본 정렬은 기존 dueDate 오름차순을 유지한다.
- 연체 항목은 색상과 텍스트로 명확히 구분한다.

**포함 범위**

- `/occurrences` 리스트의 시각적 긴급도 표시.
- 기존 `OccurrencePanel` 처리 흐름 유지.

**제외 범위**

- 드래그 정렬, 사용자 지정 우선순위. 이유는 실제 결제일 기반 자동 분류가 이번 기능의 핵심이다.

### 3. 빠른 완료 처리 진입

리마인더에서 발견한 항목을 바로 처리할 수 있도록 리스트와 대시보드에서 기존 처리 패널 진입을 강화한다.

**동작 방식**

- `/occurrences` 리스트에서 `SCHEDULED` 항목은 `처리` 버튼 텍스트를 유지하되 긴급 배지 옆에 배치한다.
- 대시보드 리마인더 항목은 상세 리스트 필터로 이동한다.
- 실제 금액이 예상 금액과 같을 가능성이 높은 항목은 기존 `OccurrencePanel`에서 기본값을 예상 금액으로 사용할 수 있게 유지한다.

**포함 범위**

- 기존 상세 처리 패널 재사용.
- 대시보드 → 리스트 → 패널로 이어지는 흐름 정돈.

**제외 범위**

- 대시보드에서 원클릭 완료. 이유는 실제 금액과 메모 입력이 필요한 케이스가 있어 오입력 위험이 있다.

## 입출력

**입력**

- 오늘 날짜. `todayIso()` 또는 동일한 `Asia/Seoul` 기준 날짜 문자열.
- occurrence 목록. `ExpenseOccurrence[]`.
- 상태 필터. `SCHEDULED`, `PAID`, `SKIPPED`.

**출력**

- 리마인더 요약. `{ overdue, today, within3Days, within7Days }` 형태의 count와 합계 금액.
- 리마인더 배지. `OVERDUE`, `TODAY`, `DUE_SOON_3`, `DUE_SOON_7`, `NORMAL` 중 하나.
- 필터 링크. `/occurrences?from=...&to=...&status=SCHEDULED`.

## 제약 조건

- 외부 네트워크 호출을 추가하지 않는다.
- 신규 DB 테이블을 만들지 않는다.
- 기존 `ExpenseOccurrence`와 `RecurringExpense` 모델을 재사용한다.
- 날짜 계산은 `Asia/Seoul` 기준 날짜 문자열로 수행한다.
- 대시보드 초기 렌더에서 API 호출 수를 과도하게 늘리지 않는다. 기존 occurrence 조회를 재사용하거나 최대 1회 추가 조회로 제한한다.

## 예외 케이스

- 예정 항목이 0건 → 리마인더 영역은 빈 상태 문구와 `정기 지출 추가` 또는 `결제 리스트` 진입을 제공한다.
- 연체 항목이 여러 개 → 가장 오래된 dueDate부터 표시한다.
- 비활성 마스터의 과거 `PAID`/`SKIPPED` occurrence → 리마인더 대상이 아니다.
- `SCHEDULED`인데 dueDate가 과거인 항목 → 연체로 표시한다.
- 날짜 파싱 실패 데이터 → 화면이 깨지지 않도록 일반 예정으로 폴백하고 API/테스트에서 정상 데이터만 생성한다.

## 채택 근거

**핵심 이유**

- 앱을 열었을 때 "지금 처리해야 할 지출"을 즉시 알게 해 정기 지출 관리의 핵심 가치를 강화한다.

**보조 이유**

- 기존 occurrence API와 화면을 재사용해 구현 비용이 낮다.
- 외부 알림 권한 없이도 체감 효용이 크다.
- 향후 브라우저 알림이나 캘린더 연동으로 확장할 수 있는 분류 로직을 먼저 만든다.

**기각된 대안**

- 브라우저 push notification부터 구현. 권한 요청, 서비스 워커, 백그라운드 동작, 알림 중복 방지 정책이 필요해 초기 범위가 커진다.
- 백엔드 스케줄러와 알림 테이블 추가. 로컬 앱 내부 시각 리마인더만으로 첫 가치를 검증할 수 있어 과하다.
- 이메일/문자 알림. 외부 네트워크 호출 없음 제약과 맞지 않는다.

## 비기능 요건

**성능**

- 대시보드 리마인더 계산은 occurrence 1,000건 기준 브라우저에서 20ms 이내.
- `/occurrences` 리스트 배지 계산은 렌더 시 단일 pass로 수행한다.

**보안**

- 위협 모델. 리마인더는 기존 지출 이름, 금액, 날짜만 표시하며 새로운 민감 데이터 저장소를 만들지 않는다.
- 외부 알림, 외부 API, 외부 저장소를 추가하지 않는다.

**확장성**

- 첫 버전은 고정 구간(오늘, 3일, 7일)을 사용한다.
- 사용자 지정 리마인더 일수와 브라우저 알림은 후속 기능에서 설정 모델을 추가해 확장한다.

## 용어 정의

- **리마인더.** 결제 예정일의 임박도나 연체 여부를 앱 내부에서 알려주는 시각적 신호.
- **연체.** `SCHEDULED` 상태이면서 dueDate가 오늘보다 이전인 occurrence.
- **임박.** dueDate가 오늘부터 7일 이내인 `SCHEDULED` occurrence.
- **리마인더 배지.** occurrence의 긴급도를 짧은 텍스트와 색상으로 표시하는 UI 요소.
